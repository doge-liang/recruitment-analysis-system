### Q11: 什么是断点续传？具体怎么实现的？（高频）

**参考答案要点**：
- **概念**：爬虫中断后，下次启动能从上次停止的位置继续，而不是从头开始
- **实现原理**：
  1. 在爬取过程中，定期保存"检查点"（Checkpoint）到JSON文件
  2. 检查点包含：当前关键词、城市、页码、职位数量
  3. 启动时先读取检查点，从记录的页码开始
  4. 使用 `CheckpointManager` 类封装检查点逻辑
- **代码结构**：
  ```python
  checkpoint = {
      "keyword": "大数据",
      "city": "上海",
      "current_page": 25,
      "total_jobs": 480,
      "timestamp": "2026-04-01T10:30:00"
  }
  ```

---

### Q12: 自适应限速是怎么工作的？

**参考答案要点**：
- **核心思想**：根据请求成功率动态调整延迟
- **算法逻辑**：
  - 初始延迟：3秒
  - 请求成功：延迟 × 0.9（逐步降低）
  - 请求失败：延迟 × 1.5（指数增加）
  - 最大延迟：30秒（防止无限增长）
  - 随机抖动：0.8x ~ 1.5x，模拟真人操作
- **好处**：在保证采集成功率的前提下，最大化采集效率

---

### Q13: Selenium和Requests有什么区别？为什么选择Selenium？

**参考答案要点**：
- **Requests**：
  - 优点：轻量、快速、资源占用少
  - 缺点：无法执行JavaScript，不能处理动态渲染页面
- **Selenium**：
  - 优点：真实浏览器环境，可执行JS，能处理复杂交互
  - 缺点：资源占用大，启动慢
- **选择原因**：前程无忧的职位数据是通过JavaScript动态加载的，必须使用浏览器渲染

---

### Q14: 爬取过程中遇到过哪些异常？怎么处理？

**参考答案要点**：
1. **网络超时**：设置超时时间，超时后重试3次
2. **元素未找到**：使用显式等待（WebDriverWait）
3. **页面结构变化**：捕获异常，记录日志，跳过该条数据
4. **浏览器崩溃**：捕获WebDriverException，保存检查点后退出
5. **数据解析失败**：try-except捕获，记录错误URL，继续处理下一条

### Q15: 如何保证采集数据的准确性和完整性？

**参考答案要点**：

## 四、数据库设计

### Q16: 数据库有哪些表？表之间的关系是什么？

**参考答案要点**（边说边画ER图）：
- **JobInfo（招聘信息）**：核心表，22个字段

### Q17: 为什么选择这些字段？有没有冗余字段？

**参考答案要点**：

### Q18: 如果数据量很大，你会怎么优化数据库？

**参考答案要点**：
1. **索引优化**：对查询频繁的字段加索引（如city、educational、createTime）
2. **分区**：按时间或城市对JobInfo表分区
3. **读写分离**：主库写，从库读
4. **缓存**：Redis缓存热点数据（如首页统计）
5. **归档**：旧数据迁移到历史表，减少主表数据量

### Q19: 数据库事务在哪些场景下使用？

**参考答案要点**：
- **爬虫数据入库**：批量插入时使用事务，保证原子性
- **用户操作**：注册时创建User和UserProfile，使用事务确保两者同时成功或失败
- **代码示例**：
  ```python
  from django.db import transaction
  
  @transaction.atomic
  def save_jobs(jobs):
      for job in jobs:
          JobInfo.objects.create(**job)
  ```

## 六、机器学习算法（重点）

### Q23: 为什么选择随机森林做薪资预测？（高频）

**参考答案要点**：
1. **处理非线性关系**：薪资与学历、经验之间不是简单线性关系
2. **抗过拟合**：集成多棵树，通过平均降低方差
3. **特征重要性**：可以输出特征重要性，帮助理解哪些因素影响薪资
4. **无需特征缩放**：对特征的尺度不敏感
5. **鲁棒性**：对异常值和噪声不敏感

**深入追问**：
- 随机森林的原理是什么？（Bagging + 随机特征选择）
- 和决策树有什么区别？（随机森林是多棵树的集成，更准确稳定）

---

### Q24: 模型准确率是多少？怎么评估的？

**参考答案要点**：
- **评估指标**：R²分数（决定系数）
- **具体数值**：训练集R² ≈ 0.90，测试集R² ≈ 0.85
- **解释**：测试集能解释85%的薪资变异，属于较好的模型
- **评估方法**：
  - 数据划分：80%训练，20%测试
  - 交叉验证：5折交叉验证
  - 防止过拟合：限制max_depth，设置min_samples_leaf

**如果老师问R²是什么意思**：
> "R²是决定系数，范围0-1，表示模型能解释数据变异的比例。R²=0.85意味着模型能解释85%的薪资变化，剩下的15%可能是由模型未考虑的因素（如具体技能、公司品牌等）导致。"

---

### Q25: 为什么选择KNN做岗位推荐？

**参考答案要点**：
- **可解释性强**：推荐结果可以解释为"与你相似的岗位"
- **无需训练**：惰性学习，直接基于距离度量
- **适合本项目**：数据量不大，特征维度不高
- **参数简单**：主要调K值即可

---

### Q26: KNN中的K值是怎么确定的？

**参考答案要点**：
- **交叉验证**：尝试K=3,5,7,9,11，选择在验证集上效果最好的
- **经验法则**：K = √n（n为样本数），约等于√20000 ≈ 141，但实际测试较小的K效果更好
- **最终选择**：K=5（在准确率和计算效率间平衡）

---

### Q27: 特征是怎么处理的？（高频）

**参考答案要点**：
- **特征类型**：
  - 学历：类别特征（博士、硕士、本科、大专、不限）
  - 工作经验：类别特征（1-3年、3-5年等）
  - 公司规模：类别特征（0-20人、20-99人等）
  - 城市：类别特征
- **编码方法**：Label Encoding（标签编码）
  - 将类别映射为整数（如本科→2，硕士→3）
  - 使用sklearn的LabelEncoder
- **为什么不用One-Hot**：
  - 树模型（随机森林）可以直接处理整数编码的类别特征
  - One-Hot会增加特征维度

---

### Q28: 模型是如何集成到Web系统中的？

**参考答案要点**：
- **解耦设计**：模型训练与Web服务分离
- **训练流程**：
  1. 独立运行 `salary_predictor.py` 训练模型
  2. 模型保存为 `.pkl` 文件（使用joblib序列化）
  3. 文件存储在 `ml_model/` 目录
- **预测流程**：
  1. Web服务启动时加载模型
  2. 用户请求时，将输入特征编码
  3. 调用 `model.predict()` 得到预测结果
  4. 返回给前端展示

---

### Q29: 如果模型效果不好，你会怎么改进？

**参考答案要点**：
1. **特征工程**：
   - 增加更多特征（技能标签、公司融资状态等）
   - 特征组合（学历+经验的交叉特征）
2. **算法升级**：
   - 尝试XGBoost、LightGBM等梯度提升树
   - 尝试神经网络（但小数据量可能不如树模型）
3. **数据增强**：
   - 采集更多数据
   - 数据清洗，去除异常值
4. **集成学习**：
   - 融合多个模型的预测结果

---

### Q30: 什么是过拟合？你的项目怎么防止过拟合？

**参考答案要点**：
- **定义**：模型在训练集表现很好，但在测试集表现差，说明模型记住了训练数据而不是学习规律
- **防止方法**：
  - **限制树的深度**（max_depth=15）：防止树过于复杂
  - **叶子节点最小样本数**（min_samples_leaf=5）：防止过细的分裂
  - **交叉验证**：评估模型泛化能力
  - **训练/测试分离**：确保测试集不参与训练

## 十一、通用技术问题

### Q42: 什么是RESTful API？你的API符合RESTful规范吗？

**参考答案要点**：
- **RESTful原则**：
  - 使用HTTP方法表示操作（GET查询、POST创建、PUT更新、DELETE删除）
  - URL使用名词表示资源
  - 无状态（每次请求包含所有必要信息）
- **本项目情况**：
  - 部分符合：使用GET/POST，返回JSON
  - 不完全符合：URL设计不够规范（如 `/get_salary_avg/`）
- **改进方向**：使用Django REST Framework规范化API

---

### Q43: 什么是MVC/MTV模式？

**参考答案要点**：
- **MVC**（Model-View-Controller）：
  - Model：数据模型
  - View：视图展示
  - Controller：业务逻辑控制
- **MTV**（Model-Template-View，Django特有）：
  - Model：数据模型（同MVC）
  - Template：模板（相当于MVC的View）
  - View：视图函数（相当于MVC的Controller）
- **对应关系**：
  - Django的Template = MVC的View
  - Django的View = MVC的Controller

---

### Q44: 什么是CSRF攻击？Django怎么防护？

**参考答案要点**：
- **CSRF**（跨站请求伪造）：
  - 攻击者诱导用户访问恶意网站
  - 恶意网站自动向目标网站发送请求（如转账、修改密码）
  - 因为用户已登录，请求会携带Cookie被服务器信任
- **Django防护**：
  - 表单中添加 `{% csrf_token %}`
  - 生成随机Token，后端验证
  - AJAX请求需要添加 `X-CSRFToken` 头

---

### Q45: 你对Python的装饰器了解吗？项目中哪里用了？

**参考答案要点**：
- **概念**：装饰器是在不修改原函数代码的情况下，增加额外功能的语法糖
- **项目中使用**：
  - `@login_required`：要求登录才能访问
  - `@csrf_exempt`：豁免CSRF验证（测试用）
- **自定义装饰器**：可以给视图函数添加日志、权限检查等

---

### Q46: 什么是Docker？为什么要用Docker部署MySQL？

**参考答案要点**：
- **Docker**：容器化技术，将应用和环境打包成镜像，保证"一次构建，到处运行"
- **使用原因**：
  - 环境一致性：开发环境和生产环境完全一致
  - 快速部署：一条命令启动MySQL
  - 易于迁移：数据卷持久化，容器可重建
  - 隔离性：不影响宿主机其他服务

- **使用流程**：

  2. `git add` 添加文件到暂存区
  3. `git commit` 提交更改

- **分支管理**：
  - `main/master`：稳定版本
